There are two situations, and they install differently. Pick the one that matches
you.

<CardGroup cols={2}>
  <Card title="New project" icon="sparkles" href="#new-project">
    The thing you're building *is* the clone. Synclair and your product share one repo.
  </Card>
  <Card title="Existing app" icon="layer-group" href="#beside-an-existing-app">
    An app already exists. Synclair sits *beside* it as a companion, documenting it.
  </Card>
</CardGroup>

<Note>
The cleanest way to run either path is to point your AI agent at it and say *"set me
up with Synclair."* The `project-bootstrap` skill interviews you, clones the
foundation, blanks the seed, and reseeds it to a working hub. The steps below are
what it does under the hood.
</Note>

## New project

The clone becomes your project's repo. Synclair and your product live together from
commit one — this is the **embedded** topology.

<Steps>
  <Step title="Clone the foundation">
    ```bash
    npx synclair new my-product
    # front-door CLI: clones the foundation into ./my-product
    ```
    You can also `git clone` the foundation repo directly. Either way, the foundation
    is kept as a git `upstream` remote so you can pull updates later.
  </Step>
  <Step title="Reseed the brand & identity">
    Blank the seed and reseed it: your product name, brand color ramps, theme, and
    (optionally) a domain skill. The bootstrap flow walks you through this.
  </Step>
  <Step title="Build your product in the repo">
    From here, the repo is yours. Build your app; Synclair's hub, skills, and
    guardrails travel with the code and are available to every agent working in it.
  </Step>
</Steps>

## Beside an existing app

An app already exists and you don't want to disturb it. Synclair runs as a **sibling
companion clone** — a *separate* repo next to the product, on its own server. This is
the **watcher** topology, and it's the recommended way to adopt Synclair onto
existing code.

<Steps>
  <Step title="Clone Synclair as a sibling">
    ```bash
    # your apps live side by side on disk:
    #   my-app/            ← the existing product, untouched
    #   my-app-synclair/   ← the Synclair companion clone
    ```
  </Step>
  <Step title="Run it on its own port">
    Synclair runs its own dev server on **port 4100** — never inside your app. You
    browse the hub at `localhost:4100/synclair`. Your app keeps running wherever it
    already runs.
  </Step>
  <Step title="Populate the hub from the codebase">
    Point your agent at the existing repo and run the `existing-project-intake`
    flow. Its diggers survey the code, harvest the docs, extract the tokens, and
    catalog the components — so the hub reflects your real app instead of starting
    empty.
  </Step>
</Steps>

<Warning>
In watcher mode Synclair is **one-way**: it observes and documents the host app and
lands nothing in the host repo. Synclair is at `localhost:4100/synclair` — never a
route inside your app.
</Warning>

## A third option: inside an existing repo

If you specifically want one repo and one clone — so agents building in the product
repo get Synclair's skills ambiently — you can **co-locate** Synclair inside the
existing repo at `./synclair`, isolated so the two apps' builds don't collide. This
is the embedded topology applied to an existing repo. It's not the default (the
sibling model is), but it's supported when single-repo ergonomics matter more than
easy foundation sync.

## Running the hub

However you installed it, the hub runs on **port 4100** (not 3000) and mounts at
`/synclair`:

```bash
npm install
npm run dev     # http://localhost:4100/synclair
```
